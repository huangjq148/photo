# PRD：家庭相册可靠性优化与产品能力演进

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 产品名称 | Photo Management App |
| 文档版本 | V1.0 |
| 文档日期 | 2026-07-12 |
| 产品定位 | 面向个人与家庭的私密照片、视频管理与协作产品 |
| 当前阶段 | 已完成 MVP，进入可靠性提升与核心整理能力建设阶段 |
| 推荐策略 | 可靠性优先，随后建设照片整理和家庭回忆能力 |

## 2. 背景

项目目前已经支持账号注册与登录、相册管理、图片和视频上传、成员协作、邀请和权限、收藏、回收站、公开分享、媒体名称编辑、儿童相册等能力，已经具备家庭相册产品的基本闭环。

当前主要矛盾已经从“缺少基础功能”转变为以下三类问题：

1. 交付基线不稳定：主分支存在单元测试失败、Lint 警告和调试代码残留。
2. 数据可靠性不足：媒体文件与数据库无法处于同一事务，本地存储缺少巡检、备份与恢复机制。
3. 大文件处理能力有限：上传请求内同步完成文件读取、图片处理和视频抽帧，存在超时、内存占用和失败恢复困难。
4. 照片整理能力不足：已有拍摄时间、GPS、checksum 等基础数据，但尚未形成重复照片、时间线和地图等用户能力。

因此，本 PRD 将后续建设拆为四个阶段，第一阶段优先建立可靠交付和数据安全基线。

## 3. 产品目标

### 3.1 总体目标

- 建立可重复、可验证的交付流程，确保主分支始终可构建、可测试。
- 降低认证信息泄露、存储数据不一致和并发上传超额风险。
- 支持大图片和视频的异步处理、失败重试和状态反馈。
- 基于现有元数据逐步补齐家庭相册的整理、浏览和分享能力。

### 3.2 第一阶段成功指标

- 主分支 `npm run verify` 连续通过，无失败测试。
- ESLint 错误和警告均为 0，CI 禁止新增警告。
- 生产日志不包含令牌内容、JWT payload、密码、用户会话版本等敏感信息。
- 存储巡检可以识别数据库缺文件、孤儿文件和存储额度误差。
- 并发上传无法突破用户存储上限。
- 媒体处理失败后可见、可重试，不产生长期不可识别的半成品数据。

## 4. 用户角色

| 角色 | 说明 | 核心权限 |
|---|---|---|
| 相册所有者 | 创建并管理家庭相册的登录用户 | 管理相册、成员、邀请、媒体、分享和相册设置 |
| 相册成员 | 被邀请加入相册的登录用户 | 根据 `can_upload`、`can_delete` 权限查看、上传或删除媒体 |
| 普通登录用户 | 尚未加入目标相册的用户 | 管理自己的默认相册和已加入的相册 |
| 公开访客 | 通过有效分享链接访问的未登录用户 | 仅访问分享范围内允许公开的媒体资源 |
| 系统运维人员 | 负责部署、巡检和故障恢复 | 执行健康检查、存储审计、备份和恢复，不直接获得用户相册浏览权限 |

## 5. 用户痛点与使用场景

### 5.1 上传与处理

- 用户上传较大视频时，需要明确看到上传和处理进度。
- 处理失败时，用户需要知道失败原因并能够重试或删除记录。
- 多设备同时上传时，系统不能因为并发检查失效而突破存储额度。

### 5.2 数据安全

- 用户删除、恢复或清空回收站后，数据库记录、文件和已用空间必须保持一致。
- 运维人员需要在不读取私人媒体内容的前提下发现丢失文件、孤儿文件和额度偏差。
- 系统故障后需要能够恢复数据库与媒体文件。

### 5.3 照片整理

- 用户希望发现完全重复的照片，避免反复占用空间。
- 用户希望按拍摄时间浏览家庭成长记录。
- 用户希望在明确隐私边界的前提下按地点浏览照片。

## 6. 建设范围与阶段

### 6.1 第一阶段：可靠交付与安全基线（P0）

#### 本次包含

- 修复现有失败测试，统一媒体展示名称规则。
- 清除认证与登录流程中的敏感调试日志。
- 清理现有 ESLint 警告，并设置零警告门禁。
- 固化 lint、类型检查、单元测试、集成测试、E2E 和构建流水线。
- 增加必要的结构化错误日志，但不得输出敏感内容。
- 增加基础健康检查，区分应用、数据库和存储可用状态。

#### 本次不包含

- 对象存储迁移。
- 上传任务队列和独立 Worker。
- 时间线、地图、重复照片等新增用户功能。
- AI 标签、人脸识别和智能推荐。

### 6.2 第二阶段：存储与上传可靠性（P1）

#### 本次包含

- 数据库记录与媒体文件一致性巡检。
- 孤儿文件识别与安全清理机制。
- 用户 `storage_used` 重新核算。
- 并发上传额度原子预留与释放。
- 媒体异步处理状态机、后台任务、失败重试。
- 数据库和媒体文件备份、恢复及恢复演练文档。
- 为对象存储预留统一存储接口。

#### 本次不包含

- 多地域容灾。
- 视频多码率转码和自适应播放。
- 用户自定义存储供应商。

### 6.3 第三阶段：核心整理能力（P2）

#### 本次包含

- 基于 checksum 的完全重复媒体识别。
- 按拍摄时间组织的统一时间线。
- 批量添加至相册、修改拍摄时间、下载和删除。
- 使用游标分页替代大数据量下的深度页码分页。
- GPS 地图视图及位置隐私控制。

### 6.4 第四阶段：家庭协作与回忆（P3）

#### 后续可迭代

- 整本相册公开分享。
- 分享密码、过期时间、下载控制和访问撤销。
- 邮箱验证、密码找回和邀请邮件。
- 站内通知中心。
- 往年今日、儿童成长月报和年度精选。
- 相似照片识别、AI 标签、自然语言搜索和人脸聚类。

## 7. 功能需求

### 7.1 交付质量门禁

- 功能描述：所有合并到主分支的代码必须通过统一验证流程。
- 入口：本地 `npm run verify` 和 CI 流水线。
- 规则：
  - ESLint 必须以零错误、零警告结束。
  - 类型检查、单元测试、集成测试、E2E 和生产构建必须全部通过。
  - 任一步骤失败时终止流水线并标记失败步骤。
  - E2E 不得因为缺少测试数据而静默跳过核心场景。
- 异常：测试数据库不可用时明确提示依赖缺失，不得误报为业务测试失败。

### 7.2 安全日志治理

- 功能描述：建立统一、最小化、可追踪的服务端日志。
- 规则：
  - 禁止记录密码、Cookie、完整令牌、JWT payload、密码哈希和分享 token。
  - 用户 ID 仅在排障确有需要时以脱敏或请求关联方式记录。
  - 登录失败统一记录错误类型，不区分“用户不存在”和“密码错误”。
  - 生产日志不得包含调试分支和手动令牌解码逻辑。
  - 每个请求可使用 request ID 关联错误链路。

### 7.3 健康检查

- 功能描述：为部署和监控提供服务健康状态。
- 入口：仅用于监控的健康检查接口。
- 返回状态：
  - `healthy`：应用、数据库和存储均可用。
  - `degraded`：应用可用，但非核心依赖异常。
  - `unhealthy`：数据库不可连接或核心存储不可用。
- 规则：响应不得包含数据库连接串、磁盘绝对路径、密钥或异常堆栈。

### 7.4 存储一致性巡检

- 功能描述：检查数据库媒体记录与实际文件之间的一致性。
- 检查类型：
  - 数据库记录存在，但 original、preview 或 thumbnail 文件缺失。
  - 文件存在，但无数据库媒体记录。
  - 数据库媒体大小汇总与用户 `storage_used` 不一致。
  - 处理状态长期停留在 `pending` 或 `processing`。
- 规则：
  - 默认只读审计，不自动删除或修改数据。
  - 修复操作必须显式触发并输出操作报告。
  - 孤儿文件需经过保留期后才能删除。
  - 报告只包含 ID、文件类型、大小和状态，不输出媒体内容。

### 7.5 存储额度预留

- 功能描述：保证并发上传不会突破用户额度。
- 核心流程：
  1. 上传开始前原子检查剩余额度并预留空间。
  2. 预留成功后接收和处理文件。
  3. 上传成功后将预留转为实际已使用空间。
  4. 上传失败或超时后释放预留空间。
- 规则：
  - 预留操作必须具备唯一上传标识，防止重复扣减。
  - 超时预留可由后台任务安全回收。
  - 用户看到的可用空间应扣除尚未完成的预留。

### 7.6 异步媒体处理

- 功能描述：将耗时的图片和视频处理从上传请求中拆出。
- 状态流转：`pending → processing → normal`；异常时进入 `failed`。
- 核心流程：
  1. 原文件写入持久存储。
  2. 创建 `pending` 媒体记录并返回上传成功。
  3. Worker 提取元数据并生成预览图、缩略图。
  4. 成功后更新为 `normal`；失败后更新为 `failed` 并保存可公开的错误类型。
- 用户操作：查看处理状态、重试失败任务、删除失败媒体。
- 规则：
  - 同一任务重复执行不能生成重复媒体记录。
  - Worker 崩溃后任务能够重新领取。
  - 原文件成功持久化前，不向用户返回上传完成。

### 7.7 完全重复媒体识别

- 功能描述：识别 checksum 相同的媒体，帮助用户释放空间。
- 入口：存储空间页面或媒体库“重复项目”。
- 规则：
  - 第一阶段只识别 checksum 完全一致的文件，不判断视觉相似。
  - 默认保留最早上传或被更多相册引用的媒体。
  - 删除前展示文件名、大小、上传者、上传时间和所在相册。
  - 用户必须主动确认，不自动删除。
  - 无删除权限的媒体只能查看，不能处理。

### 7.8 统一时间线

- 功能描述：跨相册按拍摄时间浏览用户有权访问的媒体。
- 入口：主导航“时间线”。
- 展示：按年、月、日分组，支持图片和视频混排。
- 排序规则：优先 `taken_at`，缺失时使用 `uploaded_at`，相同时间使用 ID 保证稳定排序。
- 操作：预览、收藏、批量添加至相册、下载、删除。
- 分页：基于时间与 ID 的游标分页。

### 7.9 地图视图

- 功能描述：展示包含 GPS 信息且用户有权访问的媒体。
- 入口：媒体库“地图”。
- 规则：
  - 地图只在登录后展示。
  - 公开分享默认剥离或隐藏精确位置。
  - 用户可关闭地图功能或隐藏单个媒体的位置。
  - 密集区域使用聚合点，避免一次加载全部媒体。

## 11. 异常场景

| 场景 | 系统处理 | 用户提示 |
|---|---|---|
| 存储空间不足 | 不创建额度预留，不接收文件 | “存储空间不足，请清理后重试” |
| 上传中网络中断 | 标记任务中断并在超时后释放预留 | “上传未完成，可重新上传” |
| 图片解析失败 | 保留失败状态和错误分类 | “无法处理该图片，请检查文件格式” |
| FFmpeg 处理失败 | 媒体进入 failed，可重试 | “视频处理失败，请稍后重试” |
| 数据库有记录但原文件缺失 | 巡检标记异常，不自动删除记录 | 管理侧显示“原文件缺失” |
| 孤儿文件被发现 | 进入保留期，不立即删除 | 不面向普通用户展示 |
| 额度核算不一致 | 生成审计项，人工或受控任务修复 | 管理侧显示差异值 |
| 重复媒体无删除权限 | 禁用删除，只允许查看 | “你没有删除该媒体的权限” |
| 拍摄时间缺失 | 使用上传时间进入时间线 | 不额外报错 |
| GPS 权限被关闭 | 不在地图展示该媒体 | “该媒体的位置已隐藏” |
| 分享失效 | 拒绝媒体及所有变体访问 | “分享已失效或已被撤销” |

## 12. 非功能需求

### 12.1 性能

- 上传接口不得在应用内永久保留大文件 Buffer；异步方案落地后应采用流式传输或直传存储。
- 普通媒体列表首屏查询目标响应时间不超过 500ms，不包含媒体文件下载时间。
- 时间线单次返回不超过 50 条媒体，必须使用稳定游标。
- 地图聚合查询不得一次返回用户全部媒体明细。

### 12.2 安全

- 所有私有媒体变体均需经过身份和相册权限校验。
- 分享撤销或过期后，original、preview、thumbnail 均不可继续访问。
- 日志、健康检查和审计报告不得泄露密钥、令牌、绝对存储路径和媒体内容。
- GPS 默认属于私密数据，公开分享默认不暴露。

### 12.3 可用性

- 上传成功、处理中、失败和可重试状态必须视觉可区分，并提供文字说明。
- 所有失败操作必须保留用户已完成的其他成功操作。
- 批量操作需显示选中数量，并在不可逆删除前二次确认。

### 12.4 兼容性

- 支持当前项目定义的主流图片、GIF 和视频格式。
- Web 端优先支持最新版 Chromium、Safari 和 Firefox。
- 移动端页面必须支持触控、竖屏和安全区域。

### 12.5 可观测性与恢复

- 后台任务需要记录任务类型、状态、耗时和错误分类。
- 备份必须覆盖 PostgreSQL 数据、原文件及必要的派生文件策略。
- 每季度至少执行一次恢复演练，并记录恢复时间和数据完整性结果。

## 13. 验收标准

### 13.1 第一阶段

- [ ] 更新媒体名称相关规则后，现有单元测试全部通过，测试明确覆盖有自定义名称和无自定义名称两种情况。
- [ ] 执行 `npm run verify` 时，lint、类型检查、单元测试、集成测试、E2E 和生产构建全部成功。
- [ ] ESLint 输出为 0 errors、0 warnings；人为增加未使用变量时 CI 必须失败。
- [ ] 登录成功、登录失败、无 Cookie、令牌无效和退出所有设备场景下，日志均不包含令牌、payload、密码和 session version。
- [ ] 核心 E2E 用例拥有独立测试数据，不因“没有照片”而跳过。
- [ ] 健康检查在数据库断开时返回 unhealthy，且响应不包含连接串和异常堆栈。

### 13.2 第二阶段

- [ ] 构造数据库记录存在但文件缺失的场景，巡检能够准确报告媒体 ID 和缺失变体，且不自动删除记录。
- [ ] 构造无数据库记录的文件，巡检能够标记为孤儿文件，首次巡检不得立即删除。
- [ ] 人为修改 `storage_used` 后，审计能够给出数据库值、实际汇总值和差异。
- [ ] 两个并发上传请求的总大小超过剩余额度时，最多只有符合额度的一方能够成功预留。
- [ ] 上传失败或预留超时后，预留额度被释放且用户可用空间恢复。
- [ ] Worker 重复执行同一任务时，不创建重复媒体记录，不重复扣减空间。
- [ ] 视频处理失败后页面显示失败状态；用户重试成功后状态更新为 normal。
- [ ] 完成一次从备份恢复数据库和媒体文件的演练，并通过抽样文件校验。

### 13.3 第三阶段

- [ ] 上传两个 checksum 相同的文件后，重复项目页面将其归为同一组。
- [ ] 删除重复项前，页面展示大小、时间、上传者和所在相册，并要求二次确认。
- [ ] 时间线按 `taken_at DESC, id DESC` 稳定排序；拍摄时间缺失的媒体按上传时间展示。
- [ ] 在翻页期间新增媒体，不造成已加载列表的重复项。
- [ ] 带 GPS 的媒体出现在地图中；关闭位置展示后，该媒体不再出现在地图和公开分享数据中。

## 14. 里程碑建议

| 里程碑 | 主要交付 | 建议周期 | 前置依赖 |
|---|---|---:|---|
| M1 交付基线 | 测试恢复、零警告、安全日志、健康检查 | 1 周 | 无 |
| M2 存储可靠性 | 巡检、额度核算、备份恢复 | 1–2 周 | M1 |
| M3 异步处理 | 上传任务、Worker、失败重试 | 2–3 周 | M1、M2 |
| M4 整理能力 | 重复项、时间线、游标分页 | 2–3 周 | M3 |
| M5 地图与协作 | 地图隐私、相册分享、通知 | 2–3 周 | M4 |

周期仅用于排序和资源评估，实施前需要结合开发人数和部署环境拆分技术计划。

## 15. 风险与依赖

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 本地文件与数据库无法原子提交 | 产生孤儿文件或缺失记录 | 上传任务、幂等写入、一致性巡检和补偿操作 |
| 大视频占用内存和请求线程 | 上传超时或服务不可用 | 流式上传、异步任务、文件大小和并发限制 |
| 存储额度并发竞争 | 用户突破配额 | 原子预留、唯一任务 ID、超时回收 |
| GPS 泄露家庭位置 | 严重隐私风险 | 默认私密、公开分享剥离、单媒体隐藏位置 |
| 重复项误删 | 用户数据不可恢复 | 第一阶段仅 checksum 完全一致、默认保留、进入回收站 |
| 时间线时区不一致 | 照片日期分组错误 | 明确拍摄时间解析和用户时区规则 |
| 任务队列引入新运维组件 | 部署复杂度上升 | 先抽象任务接口，再根据部署规模选择数据库队列或专用队列 |

## 16. 默认假设与待确认项

### 默认假设

- 产品继续采用私有部署或单组织部署，不在近期引入计费和多租户套餐。
- 第一阶段优先解决可靠性，不改变现有页面整体视觉风格。
- 删除重复媒体仍进入现有回收站，不直接物理删除。
- 后台巡检首先以 CLI 或定时任务提供，不立即建设完整管理后台。
- 对象存储可在第二阶段设计，但实际迁移可根据部署规模独立排期。

### 实施前待确认

- 生产部署是单机、本地 NAS，还是云服务器。
- 是否允许引入 Redis、消息队列或独立 Worker 进程。
- 数据库和媒体文件的目标备份周期、保留周期及可接受恢复时间。
- 地图服务采用自托管方案还是第三方地图服务。
- 邮件服务供应商及邮件送达要求。

## 17. 明确不纳入近期范围

- 自动删除用户媒体。
- 未经用户确认的人脸识别。
- 将用户原图用于外部 AI 模型训练。
- 照片编辑器、滤镜和复杂视频剪辑。
- 直播、动态广场、评论和公开社交关系。
- 商业化订阅、支付和多级套餐。
