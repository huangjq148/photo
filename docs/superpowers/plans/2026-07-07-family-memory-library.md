# Family Memory Library Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the first implementation of the family memory library: a time-grouped family memory stream plus complete video upload, processing, playback, sharing, and quota support.

**Architecture:** Keep the existing `Album`/`AlbumMember` permission model and the existing `Photo` table as the transitional media table. Add media fields, media-oriented library functions, and a separate worker that processes pending videos with `ffprobe`/`ffmpeg`. Keep `/api/photos` compatibility while new UI and new capabilities use media-oriented naming where practical.

**Tech Stack:** Next.js App Router, React 19, Prisma, PostgreSQL, Vitest, local disk storage, `sharp`, `ffmpeg`, `ffprobe`, `tsx`.

```ts
import type { MediaFileLike, MediaType } from "@/lib/media/types";

const IMAGE_MIME_TYPES = new Set(["image/jpeg", "image/png", "image/webp"]);
const VIDEO_MIME_TYPES = new Set(["video/mp4", "video/quicktime", "video/webm"]);

export function detectMediaType(mimeType: string): MediaType {
  if (IMAGE_MIME_TYPES.has(mimeType)) return "image";
  if (VIDEO_MIME_TYPES.has(mimeType)) return "video";
  throw new Error("不支持的媒体格式");
}

export function validateMediaFile(
  file: MediaFileLike,
  limits: { maxImageMb: number; maxVideoMb: number }
) {
  const mediaType = detectMediaType(file.type);
  const maxMb = mediaType === "image" ? limits.maxImageMb : limits.maxVideoMb;
  const label = mediaType === "image" ? "图片" : "视频";

  if (file.size > maxMb * 1024 * 1024) {
    throw new Error(`${label}大小超过${maxMb}MB限制`);
  }

  return mediaType;
}
```

- [x] **Step 4: Implement transactional claim**

In `lib/media/worker/claim.ts`, implement a claim function using PostgreSQL row locking:

```sql
WITH next_job AS (
  SELECT id
  FROM "Photo"
  WHERE media_type = 'video'
    AND processing_status = 'pending'
  ORDER BY uploaded_at ASC
  FOR UPDATE SKIP LOCKED
  LIMIT 1
)
UPDATE "Photo"
SET processing_status = 'processing',
    processing_error = NULL,
    updated_at = NOW()
WHERE id IN (SELECT id FROM next_job)
RETURNING *;
```

Implementation requirements:

- Run the raw SQL inside `prisma.$transaction`.
- Return `null` when no row is returned.
- Do not use `findFirst` followed by `update` in a normal Prisma transaction; that can race if two worker processes start.
- Keep `MEDIA_WORKER_CONCURRENCY=1` by default, but the claim must still prevent duplicate processing if a second process is accidentally started.

## Implementation Notes

### Access Control Rule

Every original, preview, thumbnail, poster, playback, share, favorite, delete, restore, and download operation must use album membership checks. Do not serve media from static public directories.

### Public Video Share Rule

Public video shares are playback-only:

- Ready video: processed playback only.
- Pending or failed video: unavailable state only.
- Never expose original video URL or original video download button.

### Transitional Naming Rule

Use `media` naming for new code. Keep `photo` naming only for compatibility with existing routes, tests, and table names.

### TDD Rule

For each task:

1. Write or update the failing test first.
2. Run the targeted test and confirm the expected failure.
3. Implement the smallest change.
4. Run targeted tests.
5. Commit.

### Verification Commands

Use these before claiming completion:

```bash
npm test
npm run build
npx prisma validate
```

If database migrations require a running PostgreSQL instance and it is unavailable, stop and report the blocker with the exact failing command.
