# 媒体库可靠性与遗留清理设计

## 目标

补齐收藏、回收站和媒体列表在数据量增长后的可用性，并移除相册化改造后残留的 Space 技术债。

## 当前问题

- 回收站前端固定请求第一页 24 条，没有加载更多。
- 收藏一次返回全部数据，没有分页、搜索或筛选。
- 数据层已有部分排序参数，但相册接口和 UI 尚未完整接入。
- 回收站没有保留期限说明和清空入口。
- 旧 Space 页面、API、模型和枚举仍存在。
- API 错误状态码依赖错误字符串判断，稳定性较差。

## 列表契约

照片、收藏和回收站统一返回：

```ts
type PageResult<T> = {
  items: T[];
  page: number;
  pageSize: number;
  total: number;
};
```

`page` 最小为 1，`pageSize` 默认为 24、最大为 50。非法参数返回 422，不静默转换为意外结果。

## 媒体筛选与排序

相册媒体支持：

- `keyword`：文件名模糊搜索。
- `mediaType`：`image` 或 `video`。
- `uploaderId`：按上传者筛选。
- `takenFrom`、`takenTo`：拍摄日期区间。
- `sortBy`：`uploadedAt`、`takenAt`、`fileName`、`size`。
- `sortOrder`：`asc` 或 `desc`。

筛选条件变化时回到第一页并取消旧请求，避免慢请求覆盖新结果。

## 回收站规则

- UI 按 `deleted_at + 30 天` 展示剩余保留时间。
- 首期不执行后台自动删除，只明确告知“计划保留 30 天”。
- 提供分页、批量恢复、批量永久删除和清空回收站。
- 永久删除继续扣减上传者的存储使用量并删除全部文件变体。

## 收藏规则

- 收藏按收藏时间倒序分页。
- 取消收藏后从当前列表移除；页面不足一页时按需加载下一条补齐。
- 收藏中的已删除媒体不展示，恢复后仍保留原收藏关系。

## 统一错误

API 返回：

```ts
type ApiError = {
  error: string;
  code: string;
  issues?: unknown[];
};
```

领域错误使用稳定 code 映射状态码，不再通过中英文错误字符串判断权限和不存在状态。

## Space 清理

删除旧 `/spaces` 页面和 `/api/spaces` 兼容接口，以及 `Space`、`SpaceMember`、`SpaceInvite` 和相关枚举。执行前通过只读迁移检查确认这些表中的有效数据都已映射到 Album 模型；发现未迁移数据时停止删除。

## 体验与无障碍

- 列表使用骨架屏保留卡片尺寸，避免布局跳动。
- 搜索输入提供可见 label，错误使用 `role=alert`。
- 无限滚动同时提供“加载更多”按钮，保证键盘和辅助技术可操作。
- 交互元素具备可见焦点和至少 44px 触控目标。

## 验收标准

- 回收站和收藏超过 24 条时可访问所有数据且不重复。
- 筛选和排序结果在分页间保持稳定。
- 清空回收站后数据库记录、文件和存储计数一致。
- 非法查询参数返回带稳定 code 的 422。
- Space 数据迁移未完成时清理过程会中止。
- 清理完成后生产构建不再输出 `/spaces` 与 `/api/spaces` 路由。

## 后续迭代

- 使用现有 checksum 增加重复文件识别。
- 使用 GPS 增加地图视图。
- 自动标签、人脸聚类和智能回忆。

## 不包含范围

- 本轮不实现后台自动清空任务。
- 不实现地图、AI 标签和重复照片合并。
- 不迁移到对象存储。
