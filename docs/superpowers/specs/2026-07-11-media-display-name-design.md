# 相册媒体名称编辑设计

## 1. 背景

当前相册网格只展示照片或视频内容，上传时的原始文件名仅出现在“媒体信息”遮罩中。用户无法在相册内为媒体补充更易理解的名称，也无法直接从网格识别相似内容。

本功能参考 iCloud 相册的轻量标题体验，在每张媒体卡片下方增加名称栏，并允许有权限的用户原位编辑名称。自定义名称只用于产品内展示，不修改原始文件名、存储对象或下载文件名。

## 2. 目标

- 让用户能在相册网格中直接识别、查看和编辑照片或视频的展示名称。
- 保留原始文件名作为文件追溯和下载依据。
- 在协作相册中提供清晰、可验证的编辑权限。
- 编辑过程不打断浏览，不要求刷新页面或进入独立管理界面。

## 3. 用户角色与权限

| 角色 | 查看媒体名称 | 编辑媒体名称 |
|---|---:|---:|
| 相册拥有者 | 是 | 是 |
| 当前媒体上传者 | 是 | 是 |
| 其他相册成员 | 是 | 否 |
| 非相册成员 | 否 | 否 |

权限按“相册拥有者或媒体上传者”判断，与成员的 `can_upload`、`can_delete` 开关无关。服务端必须独立校验权限，不能只依赖前端隐藏编辑入口。

默认相册沿用同一规则：拥有者可编辑其中的媒体名称。

## 4. MVP 范围

### 本次包含

- 照片和视频统一支持自定义展示名称。
- 相册详情网格中的名称展示与单项行内编辑。
- 名称的新增、修改和清空。
- 编辑权限、输入校验、保存状态及错误反馈。
- 媒体信息中同时区分展示名称和原始文件名。

### 本次不包含

- 修改磁盘文件名、对象存储路径或下载文件名。
- 批量重命名、自动编号和命名模板。
- 名称修改历史、撤销记录或审计日志。
- 按媒体名称搜索、筛选或排序。
- 在收藏、回收站、首页最近照片或公开分享页提供编辑入口。
- AI 自动生成名称或根据图片内容推荐名称。

### 后续可迭代

- 批量命名和自动编号。
- 在更多媒体入口展示自定义名称。
- 按自定义名称搜索。

## 5. 核心概念与数据设计

在 `Media` 上新增可空字段 `display_name`，作为跨相册一致的媒体展示名称。

| 字段 | 类型 | 必填 | 默认值 | 规则 |
|---|---|---:|---|---|
| `display_name` | String / varchar(100) | 否 | `null` | 去除首尾空格后最长 100 个 Unicode 码点；空字符串归一化为 `null` |
| `original_name` | 现有 String | 是 | 上传文件名 | 保持不变，用于文件追溯与下载命名 |

媒体可能同时出现在多个相册中，因此名称属于媒体本身，而不是 `AlbumPhoto` 关系。用户在任一有权编辑的相册中改名后，所有能看到该媒体的位置读取到相同的自定义名称。

展示名称的取值规则：

1. `display_name` 非空时，展示 `display_name`。
2. `display_name` 为空时，回退展示 `original_name`。
3. UI 需要区分“自定义名称”和“原始文件名回退”，以便编辑时空值仍表示尚未命名。

## 6. 页面结构

### 6.1 媒体卡片默认态

名称栏位于缩略图下方，属于卡片正常文档流，不覆盖照片内容。

```text
┌────────────────────────────┐
│                            │
│       照片 / 视频缩略图      │
│                            │
├────────────────────────────┤
│ 海边的日落             ✎   │  有编辑权限
└────────────────────────────┘

┌────────────────────────────┐
│       照片 / 视频缩略图      │
├────────────────────────────┤
│ IMG_20260711_1830.jpg       │  只读成员
└────────────────────────────┘
```

- 自定义名称使用主要文字色；回退显示的原始文件名使用次要文字色，提示尚未自定义命名。
- 有权限者显示编辑图标，并可点击名称文本或编辑图标进入编辑态。
- 无权限者不显示编辑图标，名称区域不使用可点击样式。
- 桌面端名称单行显示，超出宽度省略；鼠标悬停时可通过原生提示查看完整名称，键盘用户可从名称元素的可访问名称读取完整内容。
- 移动端名称最多显示两行，超出后省略；编辑入口触控区域不小于 44×44px。
- 名称栏的加入会改变卡片高度；瀑布流继续按完整卡片自然排列，不固定卡片等高。

### 6.2 行内编辑态

```text
┌────────────────────────────┐
│ [ 海边的日落____________ ] │
│                  取消  保存 │
└────────────────────────────┘
```

- 输入框获得焦点并选中现有自定义名称；若当前仅回退显示原始文件名，输入框初始值为空，原始文件名作为占位提示。
- `Enter` 保存，`Esc` 取消；输入法组合输入期间按 Enter 不触发保存。
- 点击“保存”明确提交；点击“取消”或按 Esc 恢复保存前内容。
- 失焦不自动保存，避免用户点击其他卡片或滚动时产生意外改名。若内容已变化，失焦后仍保持编辑态，直到用户保存或取消。
- 同一时间允许多张卡片分别进入编辑态，但每张卡片独立管理内容和保存状态。

### 6.3 保存状态

- 提交后输入框和操作按钮禁用，“保存”显示加载状态，防止重复提交。
- 成功后退出编辑态并原位更新名称。新增或修改时提示“名称已更新”，清空时提示“名称已清除”。
- 失败时保持编辑态和未保存内容，恢复按钮可用，并显示错误提示“名称保存失败，请重试”。
- 若服务端返回无权限，退出编辑态、恢复服务端名称，并提示“你没有权限编辑此名称”。
- 若媒体已被删除或不可见，移除对应卡片并提示“该媒体已不存在”。

### 6.4 媒体信息

媒体信息遮罩中增加“名称”字段，并保留“原始文件名”字段：

- 名称：显示自定义名称；未设置时显示“未命名”。
- 原始文件名：始终显示 `original_name`。

媒体信息保持只读，MVP 不增加第二个编辑入口。

## 7. 核心交互流程

1. 用户进入相册详情页，系统加载媒体列表及每项的 `displayName`、`originalName`、`uploaderId` 和可编辑状态。
2. 有权限者点击名称或编辑图标，卡片名称栏切换为输入态。
3. 用户输入名称；前端去除首尾空格后校验长度。
4. 用户点击保存或按 Enter，客户端提交更新请求。
5. 服务端根据请求中的相册上下文，确认用户可访问该相册且为该相册拥有者或媒体上传者，并更新 `display_name`。
6. 成功后客户端以响应中的最新值更新卡片；清空名称时恢复显示原始文件名。

## 8. 接口建议

沿用现有相册媒体接口，在相册上下文内增加更新能力：

`PATCH /api/albums/:albumId/photos/:photoId`

请求：

```json
{
  "displayName": "海边的日落"
}
```

清空名称：

```json
{
  "displayName": null
}
```

成功响应至少返回：

```json
{
  "data": {
    "id": "media-id",
    "displayName": "海边的日落",
    "originalName": "IMG_20260711_1830.jpg"
  }
}
```

接口规则：

- 字段缺失表示不修改名称；`null` 或去除首尾空格后的空字符串表示清空名称。
- 去除首尾空格后长度为 1–100 个 Unicode 码点。前后端采用相同的码点计数方式，不按 UTF-16 代码单元或 UTF-8 字节数计数；由多个码点组成的组合 emoji 按其实际码点数计算。
- 超长返回 400；无权限返回 403；媒体不存在或用户不可见返回 404。
- 服务端先确认该媒体存在于路径指定的相册，再基于该相册判断拥有者权限；不能以用户是否拥有任意一个包含该媒体的相册代替当前相册校验。
- 服务端返回持久化后的标准化值，前端以响应为准。
- 更新 `display_name` 时由现有 `updated_at` 自动记录修改时间。

相册媒体列表响应新增：

- `displayName: string | null`
- `originalName: string`
- `canEditName: boolean`

`canEditName` 由服务端根据当前用户、相册和媒体计算，前端不自行推断最终权限。

## 9. 业务规则

- 自定义名称不是唯一字段，同一相册可存在重名媒体。
- 名称允许中文、英文、数字、空格、emoji 和常见标点；只限制标准化后的字符长度。
- 名称中的 HTML 或脚本内容只能作为纯文本展示，不允许按富文本解析。
- 修改名称不改变媒体归属、封面关系、收藏、分享链接或删除状态。
- 修改名称不改变下载响应中的文件名。
- 媒体软删除后不允许通过相册详情编辑名称；恢复后保留删除前的自定义名称。
- 若媒体出现在多个相册中，只要用户在请求路径指定的当前相册具备编辑资格，即可修改媒体的全局展示名称。

## 10. 异常场景

| 场景 | 系统处理 | 提示文案 |
|---|---|---|
| 名称超过 100 个字符 | 前端阻止提交，服务端再次拒绝 | 名称最多 100 个字符 |
| 输入为空或只有空格 | 转为 `null`，保存后回退显示原始文件名 | 名称已清除 |
| 普通成员直接调用接口 | 返回 403，不修改数据 | 你没有权限编辑此名称 |
| 媒体不存在或已不可见 | 返回 404，客户端移除失效卡片 | 该媒体已不存在 |
| 保存时网络失败 | 保留编辑内容，允许重试 | 名称保存失败，请重试 |
| 连续点击保存 | 首次请求期间禁用提交 | 无额外提示 |
| 两个有权限用户并发修改 | 后一次成功写入覆盖前一次，响应返回最终写入值 | 名称已更新 |

MVP 采用最后写入生效，不增加版本号、冲突弹窗或实时同步机制。

## 11. 可访问性与响应式要求

- 编辑图标按钮提供明确的 `aria-label`，如“编辑 海边的日落 的名称”。
- 输入框具备可访问名称，并与对应媒体建立文本关联。
- 编辑、保存和取消均可仅使用键盘完成，焦点顺序符合视觉顺序。
- 保存成功后焦点返回该卡片的名称或编辑按钮；保存失败后焦点留在输入框。
- 不只依靠颜色区分只读、编辑、成功或错误状态。
- 移动端不依赖 hover 才能发现编辑入口。

## 12. 非功能需求

- 性能：名称随现有媒体列表一次返回，不为每张卡片新增独立读取请求。
- 安全：接口使用现有登录态，并在服务端校验相册可见性和编辑权限；所有名称按纯文本处理。
- 兼容性：支持项目当前定义的桌面端和移动端浏览器范围。
- 一致性：照片和视频使用相同字段、接口语义与卡片交互。

## 13. 验收标准

1. 相册详情中每张照片和视频下方均显示名称；未设置自定义名称时显示原始文件名。
2. 自定义名称与原始文件名分别存储；编辑或清空名称后，原始文件名、存储路径和下载文件名不变。
3. 相册拥有者和媒体上传者能看到编辑入口，并可新增、修改或清空名称。
4. 其他相册成员看得到名称但没有编辑入口；直接调用更新接口返回 403 且数据不变。
5. 输入合法名称并保存后，卡片无需刷新页面即可显示服务端返回的新名称。
6. 输入纯空格并保存后，`display_name` 存为 `null`，卡片恢复显示原始文件名。
7. 去除首尾空格后超过 100 个 Unicode 码点时前端无法提交，绕过前端调用接口仍返回 400；前后端对 emoji 和组合字符使用相同计数结果。
8. 保存请求进行中不能重复提交；失败时保留用户输入并允许重试。
9. 桌面端名称过长时单行省略并可查看完整内容；移动端最多显示两行且编辑入口可直接触控。
10. 用户可通过键盘进入编辑、保存和取消；成功或失败后的焦点位置符合本设计。
11. 媒体信息同时展示“名称”和“原始文件名”，未设置自定义名称时“名称”显示“未命名”。
12. 同一媒体出现在多个相册时，名称修改后各相册再次读取均得到同一名称。

## 14. 默认假设与风险

- 默认媒体名称是全局属性，而不是相册内别名；这符合现有 `Media` 为文件实体、`AlbumPhoto` 只表达关联关系的数据模型。
- 当前接口以 `/api/photos` 命名但已经承载照片和视频；本功能沿用现有路由，不在本次重命名 API。
- 多相册共享媒体时，某相册的拥有者可能不是媒体上传者，却可以通过该相册上下文修改媒体在其他相册中也会显示的名称。这是“相册拥有者可编辑”与“名称属于媒体本身”共同产生的权限结果；接口必须携带并校验当前相册 ID，避免把权限无意扩大为“任意关联相册拥有者”且无法审计请求上下文。
- 卡片新增名称栏会增加网格纵向占用，需要在实现阶段重点验证瀑布流、窄屏和长名称布局。
