# 测试与交付基线设计

## 目标

为照片管理项目建立可重复、可隔离、可在本地和 CI 中一键执行的验证基线，消除当前数据库未启动导致集成测试无法运行的问题。

## 当前问题

- `npm test` 中 15 个测试文件可通过，4 个数据库集成测试文件因 `127.0.0.1:6000` 不可达而失败。
- 测试依赖外部 PostgreSQL，但项目没有启动、迁移和关闭测试数据库的标准命令。
- `package.json` 中的 lint 命令仍使用已不适合当前 Next.js 版本的 `next lint`。
- 项目没有浏览器级核心流程验收，也没有统一 CI 流程。

## 设计决策

### 独立测试数据库

使用 Docker Compose 启动只服务于测试的 PostgreSQL，固定监听 `127.0.0.1:6000`。数据库名、用户和密码只用于测试，不复用开发环境配置。

`.env.test` 保留测试连接串；所有数据库测试在执行前检查连接目标，连接串不包含测试数据库标识时立即终止，避免误清理其他数据库。

### 命令分层

- `test:unit`：不依赖数据库的 Vitest 测试。
- `test:integration`：依赖 PostgreSQL 的数据层和 API 测试。
- `test:e2e`：Playwright 核心用户流程。
- `test:db:up`、`test:db:migrate`、`test:db:down`：管理测试数据库。
- `verify`：依次执行 lint、类型检查、全部测试和生产构建。

### 数据隔离

数据库测试继续使用唯一邮箱和 UUID，并在测试结束后按外键依赖顺序清理。测试失败也必须执行断开连接和清理逻辑。不同测试文件不得依赖执行顺序或共享业务数据。

### CI

CI 启动 PostgreSQL 服务，执行 Prisma 迁移后运行验证命令。构建和测试使用独立环境变量，存储目录指向临时目录，不写入仓库中的 `data/storage`。

## 测试层级

| 层级 | 负责内容 | 不负责内容 |
|---|---|---|
| 单元测试 | 纯函数、校验、格式化、组件静态行为 | PostgreSQL、文件系统、浏览器流程 |
| 集成测试 | Prisma 查询、权限、上传事务、分享和邀请状态 | 真实浏览器布局 |
| E2E | 登录、上传、分享、邀请、权限和回收站主流程 | 穷举全部边界组合 |
| 构建检查 | 类型、Server/Client 边界、路由生成 | 业务正确性 |

## 异常处理

- Docker 不可用时输出明确安装提示，不将连接失败误报为业务测试失败。
- 迁移失败时停止后续测试。
- 测试存储目录在套件结束后清理。
- CI 无论成功或失败都停止数据库服务并上传 Playwright 报告。

## 验收标准

- 新环境按照文档执行一组命令即可完成数据库启动、迁移和全量测试。
- 集成测试不会连接开发或生产数据库。
- `npm run verify` 能明确指出失败属于 lint、类型、测试或构建中的哪一层。
- CI 能复现本地验证步骤。
- 测试完成后不残留数据库业务数据和媒体文件。

## 不包含范围

- 性能压测平台。
- 生产环境监控和告警。
- 多数据库版本兼容矩阵。
