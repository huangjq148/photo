# 相册权限与邀请设计

## 目标

建立一致、可解释的家庭相册权限模型，并把当前直接添加已注册用户的流程升级为可接受、可过期、可撤销的邀请流程。

## 当前问题

- `updateAlbum`、`setAlbumCover`、`addAlbumMemberByEmail` 和邀请数据层只检查相册成员身份，普通成员可能修改相册或添加其他人。
- `AlbumInvite`、接受接口和 token 已存在，但 POST 邀请接口实际直接创建成员。
- 未注册邮箱无法被邀请。
- 待处理邀请没有展示、撤销和重新发送入口。

## 权限矩阵

| 操作 | owner | member + 开关 | 普通 member |
|---|---:|---:|---:|
| 查看相册和媒体 | 是 | 是 | 是 |
| 上传或向相册添加媒体 | 是 | `can_upload` | 否 |
| 从相册移除、软删除媒体 | 是 | `can_delete` | 否 |
| 编辑相册名称和描述 | 是 | 否 | 否 |
| 设置相册封面 | 是 | 否 | 否 |
| 邀请、撤销邀请、移除成员 | 是 | 否 | 否 |
| 修改成员权限 | 是 | 否 | 否 |
| 删除相册 | 是 | 否 | 否 |
| 主动退出相册 | 不允许 | 是 | 是 |

成员内容权限只控制上传和删除，不隐式获得相册管理权限。owner 不能被移除或退出；本阶段不做所有权转让。

## 邀请流程

1. owner 输入邮箱并创建 7 天有效的 pending 邀请。
2. 系统生成 token 和可复制链接，不要求目标邮箱已注册。
3. 未登录访问者被引导登录或注册，并保留邀请 token。
4. 登录邮箱必须与邀请邮箱完全一致，比较前统一转为小写。
5. 接受成功后，在一个事务中创建成员并把邀请标记为 accepted。
6. 重复接受返回已加入相册，不创建重复成员。
7. owner 可以撤销 pending 邀请，或重新发送并生成新 token；旧 token 立即失效。

## 接口设计

- `GET /api/albums/:albumId/invites`：owner 查看邀请列表。
- `POST /api/albums/:albumId/invites`：创建邀请，请求 `{ email }`。
- `DELETE /api/albums/:albumId/invites/:inviteId`：撤销 pending 邀请。
- `POST /api/albums/:albumId/invites/:inviteId/resend`：旧邀请标记 expired，创建新 token 和 7 天有效期。
- `POST /api/album-invites/:token/accept`：登录用户接受邀请，保持幂等。

重复 pending 邀请返回 409，并同时返回现有邀请的基本状态，不泄露其他相册信息。

## 页面与交互

- 成员管理仅对 owner 展示邀请表单和权限开关。
- 邀请列表展示邮箱、状态、发送时间、过期时间以及复制、重新发送、撤销动作。
- 普通成员只看到成员列表和“退出相册”。
- 邀请落地页展示相册名称、邀请人昵称和目标邮箱的脱敏形式。
- 注册或登录完成后返回原邀请页，不直接接受，仍需用户确认。

## 数据规则

- `role` 只允许 `owner`、`member`。
- pending、accepted、expired、revoked 使用枚举或集中常量，禁止任意字符串扩散。
- 同一相册、同一邮箱最多存在一个有效 pending 邀请。
- 所有成员和邀请写操作更新相册 `updated_at`。

## 验收标准

- 普通成员调用相册管理和邀请接口均返回 403。
- `can_upload` 和 `can_delete` 分别独立生效。
- 未注册邮箱可以通过注册相同邮箱后接受邀请。
- 错误邮箱、过期、撤销和重复接受都有确定结果。
- owner 可以查看、复制、撤销和重新发送 pending 邀请。
- 接受邀请并发请求不会产生重复成员。

## 不包含范围

- 相册所有权转让。
- 管理员角色和自定义权限组。
- 实际邮件发送服务；首期提供可复制邀请链接。
