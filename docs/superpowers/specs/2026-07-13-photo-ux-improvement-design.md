## 2. 背景

当前产品已经形成个人和家庭照片管理的主要功能闭环，包括相册、时间线、回忆、地图、重复媒体、收藏、回收站、公开分享、成员协作以及图片和视频预览。

现有暗色视觉风格、基础组件和功能覆盖较完整，但关键任务链路仍存在以下体验问题：

1. 用户知道自己想做什么，却难以找到正确入口。例如首页“上传照片”并不直接进入上传流程，真正上传入口位于“管理相册”二级页签。
2. 某些状态会使操作无法继续。例如搜索无结果后搜索框消失，用户无法清空关键词。
3. 桌面端依赖悬停的照片操作在触屏设备上不可发现，透明按钮还可能被误触。
4. 永久删除等不可逆操作缺少统一保护，不同页面对“移除”“删除”“永久删除”的语义不一致。
5. 大照片库场景不完整。例如“从全部照片添加”只读取前 50 张照片。
6. 加载、错误、空状态和成功反馈缺少统一规范，部分局部错误会替换整个页面。
7. 弹窗和全屏预览缺少完整的焦点管理，键盘和辅助技术用户可能失去上下文。

本设计保留现有视觉方向和业务能力，重点优化信息架构、关键任务路径、状态反馈、安全性和跨设备可用性。

## 3. 设计目标

### 3.1 产品目标

- 用户从首页发起上传时，最多经过一次目标相册确认即可开始选择文件。
- 用户可以在任何数据状态下继续操作，包括加载中、无结果、请求失败和权限不足。
- 照片的预览、选择和更多操作在桌面与触屏设备上都清晰可发现。
- “从相册移除”“移入回收站”“永久删除”具有稳定且一致的语义。
- 照片库超过 50、500 或更多项目时，搜索、筛选、选择和分页仍然可用。
- 键盘用户可完成与鼠标用户等价的核心任务。
- 用户对照片展示方式的偏好在页面切换和再次访问后保持不变。

### 3.2 成功指标

第一版不强制引入用户行为分析平台，以可验证的任务指标和可用性测试为准：

| 指标 | 目标 |
|---|---|
| 从首页到达文件投放区 | 不超过 2 次点击或轻触 |
| 搜索无结果后的恢复操作 | 1 次点击“清空搜索”即可恢复 |
| 移动端进入照片批量选择 | 入口始终可见；不依赖 hover |
| 永久删除误触保护 | 所有入口均需二次确认 |
| 已有照片选择覆盖范围 | 可访问全部结果，不受前 50 条限制 |
| 加载失败恢复 | 当前上下文内提供重试，不要求刷新页面 |
| 触控目标 | 核心操作最小 44 × 44 CSS px |
| 键盘操作 | 核心任务无焦点丢失、无键盘陷阱 |
| 布局稳定性 | 列表首屏加载使用等高骨架，主要内容不整体闪烁 |

## 4. 设计原则

1. **高频操作前置**：上传、添加和浏览是一级操作；管理资料、成员和危险操作是二级操作。
2. **状态不阻断上下文**：搜索、加载和局部保存失败不得移除用户当前页面或输入内容。
3. **危险程度决定交互成本**：可撤销操作可快速完成；不可撤销操作必须明确确认。
4. **跨设备等价**：不把关键能力仅放在 hover、右键或隐藏手势中。
5. **渐进增强**：无限滚动必须有可点击的加载更多兜底；剪贴板失败必须允许手动复制。
6. **延续现有设计系统**：保留 Noir 暗色玻璃质感、现有颜色变量和 Lucide 图标体系。
7. **偏好稳定、数据状态可分享**：个人显示偏好本地持久化；影响结果集的搜索筛选同步到 URL。

## 5. 用户角色与权限

| 角色 | 说明 | 与本设计相关的权限 |
|---|---|---|
| 未登录访客 | 浏览首页、登录注册和有效公开分享 | 不显示上传、整理、删除和账户操作 |
| 登录用户 | 管理自己的默认相册和已加入相册 | 可浏览有权访问的媒体；可管理自己的偏好 |
| 相册拥有者 | 相册创建人 | 上传、添加、从当前相册移除、设置封面、管理成员和分享；只能将自己上传的媒体移入回收站 |
| 可上传成员 | 具备上传权限的相册成员 | 上传新媒体、从本人有权访问的照片中添加 |
| 可删除成员 | 具备删除权限的相册成员 | 从当前相册移除媒体；只能将自己上传的媒体移入回收站 |
| 只读成员 | 只有查看权限的成员 | 预览、下载和收藏；隐藏编辑与删除入口 |
| 公开分享访客 | 使用分享链接查看照片或相册 | 仅执行分享配置允许的查看与下载 |

前端隐藏无权限入口只能减少误操作，所有权限仍必须由服务端校验。收到 `403` 时，保留当前上下文并提示“你的权限已发生变化，请刷新后重试”。

删除权限拆为两个互不继承的能力：

- **关系删除权**：相册拥有者或当前相册 `can_delete=true` 的成员可以从当前相册移除任意媒体关系，不影响媒体实体、上传者默认相册或其他相册。
- **媒体管理权**：只有媒体上传者可以将媒体实体移入回收站、恢复或永久删除。相册拥有者不会因为媒体被添加到自己的相册而获得媒体实体删除权。

服务端必须按以上规则收紧现有“任一关联相册拥有者可管理媒体”的宽泛判断。对于非本人上传的媒体，相册管理者只显示“从相册移除”，不显示“移入回收站”或“永久删除”。

## 7. 目标信息架构

### 7.1 一级导航

一级导航只承载高频浏览和整理入口：

| 顺序 | 入口 | 路由 | 说明 |
|---:|---|---|---|
| 1 | 相册 | `/albums` | 默认落点，管理和进入相册 |
| 2 | 时间线 | `/timeline` | 跨相册按时间浏览 |
| 3 | 回忆 | `/memory` | 往年今日、儿童成长和年度精选 |
| 4 | 收藏 | `/favorites` | 高频回看集合 |
| 5 | 更多 | 弹出菜单 | 地图、重复项目、回收站 |

“上传”不是导航目的地，应作为独立主操作按钮存在：

- 桌面端：顶部导航右侧，在用户菜单之前显示“上传”。
- 移动端：底部导航中央主按钮或顶部固定主按钮；首版推荐底部导航中央按钮。

### 7.2 移动端导航

宽度小于 `768px` 时使用固定底部导航：

- 可见项：相册、时间线、上传、回忆、更多。
- 当前项使用图标、文本和高对比背景共同表示，不只依赖颜色。
- 底部内容增加 `padding-bottom: calc(72px + env(safe-area-inset-bottom))`，避免遮挡。
- “更多”打开底部抽屉，包含收藏、地图、重复项目、回收站、账户安全和退出。
- 顶部只保留品牌、当前页面标题或返回入口，不再放置可横向滚动的七项导航。

### 7.3 桌面端导航

- 顶部导航继续 sticky，但减少同级项目。
- “更多”使用点击打开的菜单，不使用仅 hover 展示。
- 菜单支持 Escape 关闭、方向键移动和关闭后焦点恢复。
- 账户入口显示头像占位、昵称和下拉箭头；退出放入账户菜单并增加确认或进行中状态，避免作为顶栏裸按钮。

## 8. 核心用户流程

### 8.1 上传新照片

```text
首页/全局上传按钮
  → 选择目标相册（已有上下文时跳过）
  → 上传面板
  → 选择、拖拽或粘贴文件
  → 校验并加入队列
  → 开始上传
  → 单项进度与总进度
  → 成功项进入相册，失败项保留并可重试
```

### 8.2 从全部照片添加到相册

```text
相册详情“添加照片”
  → 选择“上传新照片”或“从全部照片添加”
  → 搜索/筛选全部照片
  → 跨页选择
  → 确认添加 N 项
  → 保留当前位置并刷新相册内容
```

### 8.3 搜索照片

```text
输入关键词
  → 300ms 防抖
  → 取消旧请求
  → 保留现有结果并显示局部加载
  → 展示结果或无匹配状态
  → 清空关键词恢复默认结果
```

### 8.4 删除照片

```text
单项/批量操作
  → 根据上下文选择“从当前相册移除”或“移入回收站”
  → 单项“从相册移除”直接执行；其他删除操作展示影响范围并确认
  → 执行
  → 可恢复操作显示撤销；永久删除不显示撤销
```

## 9. 功能设计

### 9.1 全局上传入口

#### 9.1.1 入口规则

- 首页“上传照片”必须直接打开上传流程，不再仅跳转 `/albums`。
- 相册详情显示两个明确动作：
  - 主按钮：“上传新照片”。
  - 次按钮：“从全部照片添加”。
- 全局上传入口在非相册页面先打开“选择目标相册”步骤。
- 从相册详情发起时自动使用当前相册，不重复询问。
- 只读成员不显示上传按钮。

#### 9.1.2 目标相册选择

- 默认选中系统“全部照片”相册。
- 展示最近使用的 5 个可上传相册，其余可搜索。
- 允许同时选择一个目标业务相册；媒体始终同时进入用户默认“全部照片”。
- 没有可上传相册时，提供“创建新相册”入口；创建成功后自动返回上传流程并选中新相册。

#### 9.1.3 导航行为

- 上传面板使用路由无关的全局弹层，不改变当前页面 URL。
- 完成后提供“查看所在相册”和“继续上传”。
- 用户从相册内完成上传后，后台刷新图库，不滚动到页面顶部。

### 9.2 上传面板与队列

#### 9.2.1 文件加入方式

- 点击文件选择器。
- 拖拽文件到投放区。
- 页面处于上传面板时粘贴剪贴板图片。
- 文件选择器保持 `multiple`，支持现有图片和视频格式。

#### 9.2.2 投放区

默认文案：

> 拖入照片或视频，或点击选择文件
>
> 支持 JPG、PNG、WebP、GIF、MP4、WebM 和 MOV

拖拽进入时提高边框和背景对比；无效文件拖入时不加入队列，并在投放区下方说明原因。

#### 9.2.3 单项状态机

```text
queued → uploading → success
                  ↘ failed → queued（重试）
                  ↘ cancelled
```

| 状态 | 用户文案 | 可用操作 |
|---|---|---|
| `queued` | 等待上传 | 移除 |
| `uploading` | 上传中 42% | 取消 |
| `success` | 上传完成 | 查看 |
| `failed` | 上传失败：原因 | 重试、移除 |
| `cancelled` | 已取消 | 重新上传、移除 |

前端不得直接展示英文内部状态值。

#### 9.2.4 列表展示

每项包含：

- 本地缩略图；视频显示播放标识和时长（可获得时）。
- 文件名、格式化文件大小。
- 进度条和状态文案。
- 与当前状态匹配的操作按钮。

完成和失败状态使用图标与文案共同表达，不只使用颜色。

#### 9.2.5 总体进度

上传开始后固定显示：

- `已完成 12 / 36`。
- 总体进度条。
- 成功、失败、取消数量。
- “取消全部”仅影响上传中和等待中的项目。
- 全部结束后提供“重试全部失败项”和“清除已完成”。

#### 9.2.6 并发与刷新

- 前端同时上传 3 个文件，避免串行上传耗时过长和无限并发占满连接。
- 同一文件项同一时刻只能存在一个请求。
- 单项成功后更新本地计数；整个批次结束或用户关闭面板后统一刷新图库。
- 不为每个成功文件重新挂载整个 `PhotoGallery`。

#### 9.2.7 关闭和离开

- 仅有等待项且尚未上传时，关闭面板需要确认“关闭后将清空待上传列表”。
- 存在上传中项目时，关闭面板需要确认“关闭会取消正在上传的项目”。
- 全部结束后可直接关闭。
- 首版上传不支持页面刷新后继续，关闭确认文案必须明确这一点。

#### 9.2.8 技术说明

现有 `fetch` 无法获得可靠的上传进度。前端需选择以下实现之一：

- 推荐：上传请求使用 `XMLHttpRequest` 的 `upload.onprogress`，其他请求继续使用 `fetch`。
- 若本期不改请求层：状态仅显示“不确定进度条”和已完成数量，不得伪造百分比。

本设计要求最终交付真实进度，实施计划应采用第一种方案。

### 9.3 相册图库工具栏

#### 9.3.1 固定结构

图库只要相册加载成功，无论照片数量和搜索结果是否为零，都必须渲染工具栏。结构顺序为：

1. 搜索框。
2. 筛选按钮。
3. 排序选择。
4. 分组选择。
5. 结果数量和清空条件。

图库卡片、骨架、空状态和错误状态位于工具栏下方，不得替换工具栏。

#### 9.3.2 搜索

- 可见标签：“搜索照片和视频”。
- 占位文案：“搜索名称或原始文件名”。
- 输入 300ms 后发起请求。
- 新请求开始前中止旧 `AbortController`。
- 输入法组合输入期间不发起搜索。
- Escape 清空当前输入；有内容时显示可点击的清除按钮。
- 关键词同步到 URL `q` 参数，刷新和后退后保持结果。
- 搜索期间保留旧结果并在结果区顶部显示细进度条或 spinner，不显示全页“加载照片”。

#### 9.3.3 筛选

首版提供：

- 媒体类型：全部、图片、视频。
- 收藏状态：全部、仅收藏。
- 拍摄时间：起始日期、结束日期。
- 上传者：仅在协作相册显示。

筛选面板显示已启用条件数量。应用后同步到 URL；“清空筛选”不清空搜索关键词。

#### 9.3.4 排序与分组

排序：

- 拍摄时间：新到旧、旧到新。
- 上传时间：新到旧、旧到新。
- 名称：A-Z、Z-A。

分组：不分组、按月、按年。分组不改变数据排序字段，只改变展示分段。

“时间倒序”不再作为分组选项，避免把排序和分组概念混在一起。

#### 9.3.5 空状态

| 场景 | 标题 | 说明 | 操作 |
|---|---|---|---|
| 相册完全为空 | 这个相册还没有照片 | 上传新照片，或从全部照片中添加 | 上传新照片、从全部照片添加 |
| 搜索无结果 | 没有找到匹配的照片 | 尝试其他关键词 | 清空搜索 |
| 筛选无结果 | 当前筛选条件没有结果 | 调整或清空筛选条件 | 清空筛选 |
| 搜索和筛选同时生效且无结果 | 没有匹配当前条件的照片 | 可分别清除关键词或筛选 | 清空全部条件 |

#### 9.3.6 错误状态

- 首次加载失败：工具栏下显示错误面板和“重新加载”。
- 加载更多失败：保留已加载项目，在列表尾部显示“加载失败，重试”。
- 搜索失败：保留旧结果，在工具栏下显示非阻断错误，不清空输入。
- `401`：跳转登录并携带返回地址。
- `403`：显示权限变化说明和刷新按钮。

### 9.4 分页与无限滚动

- 相册图库和全部照片选择器每次加载 24 项。
- 自动无限滚动保留，但列表尾部必须始终有键盘可操作的“加载更多”按钮；进入预加载范围时可自动触发同一函数。
- 自动触发和按钮点击必须共享请求锁，防止重复页。
- 结果需按稳定的排序字段加 ID 排序，防止翻页重复或遗漏。
- 搜索、筛选或排序变化时重置游标、选择状态和已加载结果。
- 加载状态使用 6～12 个与当前照片尺寸一致的骨架卡片。
- 到达结尾显示“已显示全部 N 项”，不显示只有当前加载数量的模糊统计。

### 9.5 从全部照片添加

#### 9.5.1 数据范围

- 不再固定读取第一页 50 项。
- 使用与图库相同的搜索、媒体类型和日期筛选能力。
- 数据源为当前用户的默认“全部照片”相册，查询时传入当前目标相册作为 `excludeAlbumId`。
- 默认排除已在当前相册中的项目。
- 已在目标相册中的项目不出现在选择器结果中；本期不返回或展示禁用的“已添加”卡片。

#### 9.5.2 选择规则

- 支持当前已加载结果全选。
- 选择状态跨分页保留。
- 页脚固定显示“已选择 N 项”。
- 关闭弹窗时若存在选择，确认是否放弃。
- 提交后只关闭选择器，不关闭相册页面。

#### 9.5.3 操作结果

- 全部成功：“已将 N 项添加到「相册名称」”。
- 部分成功：保留失败项目的选择并显示“成功 N 项，失败 M 项”，允许重试失败项。
- 全部失败：保留全部选择和筛选位置。
- 已由其他操作添加的项目视为幂等成功，不重复创建关系。

### 9.6 照片卡片

#### 9.6.1 操作可见性

桌面端：

- 鼠标悬停或卡片内获得焦点时显示选择与更多按钮。
- 已选择时选择按钮始终可见。
- 已打开更多菜单时按钮和菜单保持可见。

触屏端：

- 选择按钮和更多按钮始终可见。
- 按钮使用半透明深色背景，保证在浅色照片上可见。
- 不依赖长按；长按进入选择模式可作为增强能力，但不能成为唯一入口。

使用媒体查询 `@media (hover: hover) and (pointer: fine)` 控制仅桌面隐藏，而不是单纯使用 Tailwind `group-hover`。

#### 9.6.2 点击区域

- 点击照片主体：打开预览。
- 点击名称：有编辑权限时进入名称编辑；无权限时不渲染伪按钮。
- 点击选择按钮：切换选择状态，不打开预览。
- 点击更多按钮：打开操作菜单，不依赖 hover。

#### 9.6.3 更多菜单

按以下顺序分组：

1. 收藏/取消收藏、分享、下载。
2. 修改拍摄时间、位置可见性、添加到相册、设为封面。
3. 媒体信息。
4. 从相册移除或删除，使用分隔线和危险色。

菜单要求：

- 点击打开和关闭。
- 点击外部、Escape 或选择操作后关闭。
- 使用 `aria-haspopup="menu"`、`aria-expanded` 和 `role="menu"`。
- 上下方向键移动，Enter/Space 执行。
- 关闭后焦点回到更多按钮。
- 根据权限隐藏不可执行项，不显示点击后才报错的入口。

#### 9.6.4 信息面板

“媒体信息”改为非可点击遮罩式详情面板或弹窗，不再通过点击任意遮罩关闭。展示：

- 显示名称、原始文件名。
- 尺寸、格式、文件大小、视频时长。
- 拍摄时间、上传时间。
- 所在相册数量。
- 位置是否隐藏。

提供明确关闭按钮并支持 Escape。

### 9.7 选择模式与批量操作

#### 9.7.1 进入方式

- 点击任一照片选择按钮进入选择模式。
- 桌面端工具栏提供“选择”按钮。
- 移动端可额外支持长按卡片进入，但仍保留可见按钮。

#### 9.7.2 选择模式行为

- 已选择照片使用边框、勾选图标和轻微遮罩共同表示。
- 卡片主体在选择模式下默认切换选择，不打开预览；提供独立预览小按钮。
- 支持“选择当前已加载结果”和“取消全部”。
- 搜索、筛选、排序改变前，如果存在选择，提示用户继续会清空当前选择。

#### 9.7.3 批量操作栏

- 桌面端：图库工具栏下 sticky。
- 移动端：固定在底部导航上方。
- 显示“已选 N 项”、添加到相册、收藏/取消收藏、删除和取消。
- 操作进行中禁用重复提交，显示动作级 spinner。
- 批量操作失败时不得清空全部选择；成功项取消选择，失败项保持选择。

### 9.8 删除语义与确认

#### 9.8.1 三种删除语义

| 操作 | 数据影响 | 是否可撤销 | UI 文案 |
|---|---|---|---|
| 从当前相册移除 | 删除相册与媒体关系，媒体仍在全部照片和其他相册 | 可重新添加 | 从相册移除 |
| 移入回收站 | 媒体在用户端隐藏，保留恢复机会 | 可恢复 | 移入回收站 |
| 永久删除 | 删除数据库记录及文件变体 | 不可撤销 | 永久删除 |

默认相册中不显示“从相册移除”，只能“移入回收站”。普通相册中同时存在两种操作时，必须明确影响范围。

#### 9.8.2 统一确认弹窗

新增公共 `ConfirmationDialog`，属性至少包括：

```ts
type ConfirmationDialogProps = {
  open: boolean;
  title: string;
  description: ReactNode;
  confirmLabel: string;
  cancelLabel?: string;
  tone: "default" | "warning" | "danger";
  busy?: boolean;
  requireText?: string;
  onConfirm: () => void | Promise<void>;
  onClose: () => void;
};
```

规则：

- 从相册移除：批量操作需要确认；单项可直接执行并显示撤销反馈。
- 移入回收站：单项和批量均确认，说明可从回收站恢复。
- 永久删除：单项和批量均确认，标题和主按钮必须含“永久删除”。
- 清空回收站：输入“永久删除”后才能确认。
- 弹窗不得使用浏览器原生 `window.confirm`。

#### 9.8.3 撤销

- “从相册移除”和“移入回收站”成功后显示 6 秒可操作消息：“已移入回收站 · 撤销”。
- 撤销调用现有恢复或重新添加关系接口。
- 若撤销失败，提示原因并提供进入回收站或重新添加的入口。
- 永久删除不提供撤销。

### 9.9 回收站

- 个人回收站只展示当前用户上传的媒体；其他上传者删除的共享媒体会从相册中消失，但不会进入当前用户的回收站。
- 顶部持续显示总数和保留规则。
- 单项卡片显示删除日期与剩余天数；小于等于 3 天使用警告色。
- 恢复操作不需要确认。
- 单项和批量永久删除使用统一危险确认弹窗。
- “清空回收站”与其他操作保持视觉间隔，避免误触。
- 批量操作结果必须区分成功和失败，不能将错误消息显示为绿色成功通知。
- 删除或恢复进行中禁用对应项目按钮，但不阻塞其他项目浏览。

### 9.10 加载、空、错误与反馈

#### 9.10.1 加载层级

| 层级 | 使用方式 |
|---|---|
| 页面首次加载 | 页面标题立即可见，内容区使用骨架 |
| 搜索/筛选刷新 | 保留旧内容，结果区显示局部进度 |
| 加载更多 | 列表尾部骨架和“正在加载更多” |
| 单项操作 | 对应按钮 spinner，不阻塞整个页面 |
| 批量操作 | 批量栏动作级 loading，保持选择 |

#### 9.10.2 消息系统

扩展现有消息组件：

```ts
type MessageType = "success" | "error" | "warning" | "info";

type MessageOptions = {
  duration?: number;
  actionLabel?: string;
  onAction?: () => void | Promise<void>;
  persistent?: boolean;
};
```

规则：

- 成功消息默认 3 秒。
- 普通错误默认 6 秒并允许手动关闭。
- 带撤销动作的消息至少 6 秒。
- 多条消息最多同时显示 3 条，其余排队。
- 错误使用 `role="alert"`；成功和信息使用 `role="status"`。
- 相同内容在 1 秒内合并，避免批量操作刷屏。

#### 9.10.3 错误分层

- 字段错误：紧邻输入框，关联 `aria-describedby`。
- 弹窗操作错误：显示在弹窗操作区，保持输入和选择。
- 列表请求错误：显示在列表区域，提供重试。
- 全局会话错误：跳转登录或显示全局提示。
- 页面级致命错误只用于首次加载无法得到页面主体数据的场景。

相册详情必须拆分 `loadError` 与 `formError`，资料保存失败不得触发页面级错误返回。

### 9.11 公共弹窗

#### 9.11.1 焦点管理

- 打开时记录触发元素。
- 首选聚焦显式传入的 `initialFocusRef`；否则聚焦第一个可交互元素；无交互内容时聚焦弹窗容器。
- Tab 和 Shift+Tab 循环在弹窗内部。
- Escape 调用 `onClose`，但 `closeOnEscape=false` 时不关闭。
- 关闭后将焦点还给触发元素；触发元素已卸载时聚焦最近的页面标题或主区域。
- 背景内容使用 `inert` 或等价机制阻止焦点进入。

#### 9.11.2 关闭规则

公共 Modal 增加：

```ts
type ModalProps = {
  // existing props
  closeOnBackdrop?: boolean;
  closeOnEscape?: boolean;
  initialFocusRef?: RefObject<HTMLElement | null>;
  onBeforeClose?: () => boolean | Promise<boolean>;
};
```

`onBeforeClose` 返回 `true` 表示允许关闭，返回 `false` 表示保持打开；未传时直接关闭。

- 信息类弹窗默认允许点击遮罩和 Escape 关闭。
- 有未保存表单、选择或上传任务的弹窗实现 `onBeforeClose`。
- 操作进行中默认禁止关闭，除非操作明确支持取消。
- 移动端弹窗主体最大高度使用 `100dvh` 和安全区，页头、内容、页脚分别固定、滚动、固定。

#### 9.11.3 表单提交

- 创建相册、编辑资料等内容使用 `<form>`，Enter 提交，按钮 `type="submit"`。
- 多行文本框中的 Enter 不提交。
- 创建弹窗打开后自动聚焦相册名称。
- 保存失败保留全部字段和当前页签。

### 9.12 图片和视频预览

- 全屏预览复用公共焦点约束和焦点恢复逻辑。
- 打开后焦点放在关闭按钮或预览容器。
- 左右方向键切换；Escape 关闭；`+`、`-` 控制缩放；`0` 恢复适应视图。
- 移动端上一张、下一张按钮距离屏幕边缘至少 8px，不被安全区遮挡。
- 触摸手势：单指横滑切换；双指缩放；图片放大后单指拖动。不得用全局 `touch-action: none` 阻断未放大时的横滑。
- 底部信息条在窄屏限制宽度并允许名称截断，不遮挡媒体主体。
- 预览导航到当前已加载集合末尾且列表仍有更多数据时，触发加载下一页并继续导航；失败时保留当前媒体并提供重试。
- 视频切换离开当前项时暂停播放，避免后台继续出声。
- 图片和视频使用相同的关闭、序号和标题结构。

### 9.13 用户显示偏好

#### 9.13.1 偏好项

```ts
type GalleryPreferences = {
  photoSize: "small" | "medium" | "large";
  layoutMode: "grid" | "waterfall";
  showTakenAt: boolean;
  groupMode: "none" | "month" | "year";
};
```

#### 9.13.2 存储规则

- 首版保存到 `localStorage`，键名 `photo:gallery-preferences:v1`。
- 默认值：中等尺寸、网格、隐藏时间、不分组。
- 初始化时在客户端读取，读取失败或结构无效时回退默认值。
- 使用版本号，后续字段变化可迁移或重置。
- 搜索、筛选和排序不是显示偏好，写入 URL 而不是 `localStorage`。
- 偏好对所有相册生效，不按相册分别保存。

#### 9.13.3 控件位置

- 桌面端在图库工具栏的“显示设置”弹出面板中调整。
- 移动端在浮动工具或工具栏的“显示”入口中调整。
- 当前设置摘要应可见，例如“网格 · 中 · 显示时间”。

### 9.14 首页与相册列表

- 首页快捷操作中的“上传照片”打开全局上传流程。
- 相册卡片的空封面使用 Lucide `Image` 或 `Images` 图标，不使用 emoji。
- 整张相册卡片可点击进入详情；信息按钮保持可见且有明确点击反馈。
- 相册卡片信息层在移动端用点击切换，桌面端也必须支持点击，不只依赖 hover。
- 空相册列表状态提供“创建第一个相册”，创建成功后直接进入相册并提示上传。
- 最近照片点击使用统一预览器，不直接打开原图 URL。

### 9.15 认证与账户反馈

本次不重构认证业务，但统一以下操作体验：

- 登录、注册成功按钮保持 loading，导航完成前防止重复提交。
- API 字段错误映射到对应输入框，不将多个校验错误拼接成一个段落。
- 密码字段提供显示/隐藏按钮，并保留可访问名称。
- 退出登录按钮进入进行中状态；请求失败时不跳转登录页。
- “退出所有设备”使用统一危险确认弹窗，不使用原生 `confirm`。

## 11. 状态与数据模型

### 11.1 图库查询状态

```ts
type GalleryQuery = {
  q: string;
  mediaType: "all" | "image" | "video";
  favorite: "all" | "only";
  uploaderId?: string;
  takenFrom?: string;
  takenTo?: string;
  sortBy: "takenAt" | "uploadedAt" | "fileName";
  sortOrder: "asc" | "desc";
};
```

### 11.2 异步列表状态

```ts
type AsyncListState<T> = {
  items: T[];
  initialLoading: boolean;
  refreshing: boolean;
  loadingMore: boolean;
  error: string | null;
  loadMoreError: string | null;
  nextCursor: string | null;
  total: number;
};
```

不得使用单一 `loading` 同时代表首次加载、搜索刷新和加载更多。

### 11.3 选择状态

```ts
type SelectionState = {
  selectedIds: Set<string>;
  mode: "idle" | "selecting";
  pendingAction: null | "add" | "favorite" | "remove" | "trash";
};
```

选择使用 `Set`，避免在大量项目下重复线性查找。提交 API 前转换为数组。

## 12. API 设计

### 12.1 相册照片查询

`GET /api/albums/:albumId/photos`

查询参数：

| 参数 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `cursor` | string | 无 | 下一页游标 |
| `pageSize` | number | 24 | 1～50 |
| `q` | string | 空 | 显示名称和原始文件名模糊匹配 |
| `mediaType` | `image\|video` | 无 | 媒体类型 |
| `favorite` | boolean | 无 | 仅收藏 |
| `uploaderId` | string | 无 | 上传者 |
| `takenFrom` | ISO date | 无 | 拍摄起始日期 |
| `takenToExclusive` | ISO datetime | 无 | 用户所选结束日期下一日零点，作为排他上界 |
| `sortBy` | enum | `takenAt` | 排序字段 |
| `sortOrder` | `asc\|desc` | `desc` | 排序方向 |
| `excludeAlbumId` | string | 无 | 选择器中排除已属于目标相册的媒体 |

响应：

```ts
type CursorPage<T> = {
  items: T[];
  nextCursor: string | null;
  total: number;
};
```

规则：

- 排序字段相同的项目用媒体 ID 作为稳定次排序键。
- 非法参数返回 `422` 和稳定错误 code。
- 搜索关键词首尾空白由服务端清理，空字符串等同未传。
- `takenAt` 缺失时按 `uploadedAt` 参与时间排序。
- 日期筛选以用户浏览器本地时区解释：起始日期转换为当日 `00:00:00` 的 ISO 时刻并传为 `takenFrom`，结束日期转换为下一日 `00:00:00` 的 ISO 时刻并传为 `takenToExclusive`；服务端使用 `>= takenFrom` 且 `< takenToExclusive`，避免结束日遗漏和跨时区歧义。

### 12.2 批量添加到相册

`POST /api/albums/:albumId/photos`

请求：

```ts
{ photoIds: string[] }
```

响应必须支持部分结果：

```ts
type BatchMutationResult = {
  succeededIds: string[];
  failed: Array<{
    id: string;
    code: string;
    message: string;
  }>;
};
```

- 已存在的关系计入成功，保证幂等。
- 单次最多 200 项；超过返回 `422`。

`BatchMutationResult` 是所有批量媒体修改的统一响应，不只用于添加。每个输入 ID 必须且只能出现在 `succeededIds` 或 `failed` 中；HTTP `200` 表示服务端已处理整批，即使其中存在单项失败。只有请求结构非法、未登录或整个目标资源不可访问时使用 4xx，并且不返回部分结果。

### 12.3 从相册移除

- 单项：`DELETE /api/albums/:albumId/photos/:photoId`。
- 批量：现有批量接口需确认语义是删除关系，而不是删除媒体；建议命名为 `POST /api/albums/:albumId/photos/batch-remove`。
- 响应使用 `BatchMutationResult`。

若为兼容性保留旧 `/batch-delete`，内部可转发，但新前端不得继续使用语义模糊的命名。

### 12.4 移入回收站与恢复

- 单项移入回收站：`DELETE /api/photos/:id`。
- 批量移入回收站：新增 `POST /api/photos/batch-trash`。
- 单项恢复：`POST /api/photos/:id/restore`。
- 批量恢复：沿用回收站批量恢复接口。
- 批量移入回收站和批量恢复均返回 `BatchMutationResult`，逐项校验媒体上传者权限和当前状态。
- 只有媒体上传者可以移入回收站和恢复；相册拥有者与 `can_delete` 成员不能处理他人上传的媒体实体。

### 12.5 收藏与取消收藏

- 单项：沿用 `POST /api/photos/:id/favorite`，请求改为显式 `{ favorited: boolean }`，避免重试切换两次。
- 批量：新增 `POST /api/photos/batch-favorite`，请求 `{ photoIds: string[], favorited: boolean }`。
- 批量接口返回 `BatchMutationResult`；对用户可访问的媒体逐项创建或删除个人收藏关系。
- 收藏是当前用户的个人关系，不需要相册删除权限；媒体不可访问时该项失败。

### 12.6 永久删除

- 单项：`DELETE /api/trash/photos/:id`。
- 批量：`POST /api/trash/photos/batch-delete`。
- 清空：`POST /api/trash/photos/clear`。

服务端必须验证媒体已处于回收站；否则返回 `409 MEDIA_NOT_TRASHED`。

- 只有媒体上传者可以永久删除。
- 批量永久删除返回 `BatchMutationResult` 并逐项执行上传者权限检查。
- “清空回收站”只处理当前用户上传且位于回收站的媒体，不删除其他上传者的媒体；响应返回 `BatchMutationResult`，前端在部分失败时保留失败项。
- 清空回收站没有客户端输入 ID；其 `BatchMutationResult` 中的 ID 并集代表服务端在请求开始时查得并尝试处理的全部当前用户回收站媒体，便于前端核对总数和测试结果完整性。

### 12.7 统一错误结构

```ts
type ApiError = {
  error: string;
  code: string;
  issues?: Array<{
    path: string;
    message: string;
  }>;
};
```

前端以 `code` 决定交互，不依赖中英文错误字符串。

## 14. 响应式设计

### 14.1 断点

验证宽度：320、375、414、768、1024、1440px。

| 宽度 | 图库默认列数（中） | 导航 | 批量操作 |
|---|---:|---|---|
| 320～767 | 2 | 底部导航 | 底部固定 |
| 768～1023 | 3 | 顶部简化导航 | 工具栏下 sticky |
| 1024～1279 | 3 | 顶部完整导航 | 工具栏下 sticky |
| 1280+ | 4 | 顶部完整导航 | 工具栏下 sticky |

小尺寸在 320px 下最多 3 列，确保照片卡片的选择和更多按钮不重叠。

### 14.2 安全区和遮挡

- 底部固定元素使用 `env(safe-area-inset-bottom)`。
- 同时出现底部导航和批量栏时，批量栏位于导航上方。
- Toast 位于顶部安全区下方；移动端占满可用宽度并保留 16px 边距。
- 全屏预览控制不得与系统手势区域重叠。

## 15. 无障碍要求

- 根布局提供“跳到主要内容”链接，目标为统一 `#main-content`。
- 每个页面只能有一个主要 `h1`；图库分组标题按层级使用 `h2` 或 `h3`。
- 搜索框拥有可见 label，不只依赖 placeholder。
- 分段选择使用 `aria-pressed` 或单选组语义；当前状态不只靠颜色。
- Switch 必须有可访问名称，并关联说明文字。
- 图片使用用户显示名称作为 alt；纯装饰图片 alt 为空。
- 图标按钮必须有具体 `aria-label`，例如“打开《夏日旅行》的更多操作”。
- 所有核心触控目标至少 44 × 44px。
- 正文和弱化文字符合 WCAG AA 对比度；危险色文字不得直接使用低对比红色。
- 动画继续遵循 `prefers-reduced-motion`。
- 加载完成、批量结果和上传进度通过合适的 live region 宣告，但进度更新频率不高于每秒一次。
- 弹窗、菜单、抽屉和全屏预览关闭后恢复焦点。

## 16. 性能要求

- 搜索防抖 300ms，旧请求必须取消。
- 首屏只请求需要的 24 项，不因预览导航一次加载全部媒体。
- 图片明确宽高比或尺寸，避免加载时布局位移。
- 列表缩略图使用 `loading="lazy"`，首屏前 4～8 张可设置高优先级。
- 优先使用 Next Image 或等价的响应式 `srcset`；GIF 和受保护媒体无法使用优化管道时保留原生图片，但必须指定尺寸。
- 上传本地预览 URL 在项目移除或组件卸载时调用 `URL.revokeObjectURL`。
- 滚动监听使用 passive 或 IntersectionObserver。
- 单页已渲染媒体达到 500 项后评估窗口化；本期不强制引入虚拟列表，但不得一次请求全部数据。
- `backdrop-filter` 在低性能设备上应有无模糊降级背景。

## 17. 安全与隐私

- 位置隐藏/显示继续需要明确确认；确认弹窗说明公开分享是否受影响。
- 公开分享默认不暴露精确 GPS 数据。
- 文件校验以服务端为准，前端 `accept` 仅用于选择提示。
- 错误消息不得暴露存储绝对路径、数据库信息或内部堆栈。
- 剪贴板粘贴只在上传面板聚焦时处理，不监听全站剪贴板。
- 永久删除必须再次校验登录状态、媒体状态和权限。

## 21. 发布与迁移策略

- 每个阶段先在相册详情启用，再迁移时间线、收藏、地图和回收站。
- API 扩展保持旧参数可用，前端迁移完成后再移除语义模糊的旧批量接口。
- `GalleryPreferences` 使用独立版本键，不读取已有未知本地字段。
- 导航改版发布时保留现有路由，不改变外部书签。
- 删除交互发布前必须完成数据层语义核对，确认“相册关系删除”和“媒体软删除”没有共用错误接口。
- 高风险变更以功能开关控制时，开关只决定新 UI，不应产生两套不同数据语义。

## 22. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 上传进度需要改变请求实现 | 进度显示无法按原 `fetch` 完成 | 单独封装 XHR 上传，不扩散到其他 API |
| 查询筛选组合增多 | 数据库查询和索引压力增加 | 为常用排序和外键补索引，限制 pageSize |
| 跨页选择状态复杂 | 部分失败后状态容易错误 | 使用统一 `useSelection` 和结构化批量响应 |
| 删除语义历史上不一致 | 可能造成误删 | 实施前逐个审计 API，补集成测试后再接 UI |
| Modal 改造影响全部弹窗 | 回归范围广 | 先补公共组件测试，再分模块迁移 |
| 底部导航遮挡内容 | 移动端操作不可达 | 统一安全区变量和页面底部间距 |
| 预览跨页增加状态耦合 | 可能重复加载或导航跳动 | 预览只通过图库数据控制器请求下一页 |

## 23. 默认决策

以下决策已在本设计中固定，开发阶段无需再次选择：

- 保留现有 Noir 暗色视觉体系。
- 上传真实进度使用 `XMLHttpRequest` 封装。
- 上传并发数固定为 3。
- 搜索防抖为 300ms。
- 列表页大小默认 24，最大 50。
- 批量提交单次上限 200 项。
- 显示偏好保存在全局 `localStorage`，不按相册区分。
- 搜索筛选写入 URL。
- 移动端使用底部导航，上传为中央主操作。
- 所有永久删除使用自定义确认弹窗；清空回收站需要输入确认文字。
- 可撤销消息显示 6 秒。
- 本次不引入新的服务端任务系统或分析平台。
