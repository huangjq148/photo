# 登录恢复设计

## 背景

当前应用代码已经依赖 `User.session_version`，但开发数据库仍有两个迁移未执行，其中 `20260710150000_add_session_version` 负责添加该字段。Prisma 在登录时查询完整用户记录，因此数据库字段缺失会导致查询异常。登录 API 当前捕获所有异常并统一返回“邮箱或密码错误”，同时把数据库异常计入登录失败限流。

## 目标

- 应用缺失迁移，使当前数据库结构与 Prisma schema 一致并恢复正常登录。
- 保留错误凭据统一返回 401 的安全行为。
- 将数据库或其他服务端异常返回为通用 500 错误，避免误导用户。
- 只有真正的凭据校验失败才计入登录限流。

## 非目标

- 不更换认证协议、密码算法或会话格式。
- 不引入 Redis 等新的限流基础设施。
- 不修改登录页面视觉设计。

## 设计

### 错误分类与响应契约

复用 `lib/api/errors.ts` 的 `AppError`，在认证领域层定义稳定的无效凭据错误 `INVALID_CREDENTIALS`。用户不存在或密码不匹配时都抛出完全相同的错误；登录路由只对该错误记录一次限流失败，并返回 `{ error: "邮箱或密码错误", code: "INVALID_CREDENTIALS" }` 和 401，避免账号枚举。

其他异常（例如 Prisma 连接失败、字段缺失或环境配置错误）不记录限流失败、不设置会话 Cookie，服务端通过 `console.error` 记录原始异常，客户端统一收到 `{ error: "登录服务暂时不可用，请稍后再试", code: "INTERNAL_ERROR" }` 和 500，不泄露内部细节。

为避免依赖不可观测的全局单例，登录路由把核心处理提取为可注入 `loginUser`、`RateLimitStore` 和日志器的处理函数；实际 `POST` 使用现有 Prisma、环境配置和单例限流器。测试使用带调用计数的 spy store 和 stub 登录函数，确定性断言计数、响应及 Cookie。

### 数据迁移

先使用显式加载 `.env.local` 的 Node 命令连接目标数据库，输出脱敏后的主机、端口、数据库名，并读取 `_prisma_migrations` 和三个废弃 Space 表的行数。由于待执行的 `20260710160000_remove_deprecated_spaces` 包含破坏性 DROP，只有 `Space`、`SpaceMember`、`SpaceInvite` 三表都存在且均为空时才允许继续；任一表非空或审计失败就停止，不执行迁移，并先制定数据备份/回填方案。运行时 Space 代码已删除，因此审计与部署之间没有合法应用写入路径。

审计通过后运行可执行命令：

```bash
node --env-file=.env.local node_modules/prisma/build/index.js migrate deploy
```

迁移后使用同样的显式环境加载方式运行 `migrate status`，确保目标数据库 schema 与代码一致。这样不会把未审计的破坏性迁移盲目绑定到登录恢复。

### 测试

先增加路由级回归测试并确认失败：

- 不存在的用户和错误密码分别返回相同的 401 body/code，并各自恰好累计一次限流失败。
- 数据库异常返回 500，记录服务端异常，不累计限流失败且不设置 Cookie。

然后实现最小错误分类逻辑，使测试通过。最后运行相关单元测试、类型检查、完整单元测试和构建，并通过实际数据库查询或登录 API 验证迁移后的登录链路。

## 验收标准

- 迁移前已确认目标数据库身份，且三个废弃 Space 表均为空；否则停止并保留全部数据。
- 审计通过并执行部署后，数据库不存在待执行迁移。
- 有效账号可以登录并获得会话 Cookie。
- 不存在的用户与错误密码返回相同的 401 body/code，各自只增加一次失败计数。
- 数据库异常返回稳定的 500 body/code，不显示为密码错误、不触发登录限流计数、不设置 Cookie，并在服务端留下原始错误日志。
- 相关测试、类型检查与构建通过。
