## 2. 一期建设目标

一期目标是完成一个可上线的照片管理应用 MVP，核心闭环如下：

```text
用户注册登录
→ 创建个人空间
→ 上传照片
→ 浏览照片
→ 管理照片
→ 删除恢复
→ 容量统计
```

共享空间、相册、批量管理、搜索筛选等能力属于一期后半段或 MVP+1 增强，不作为最小可上线闭环的前置条件。

一期重点不是做大量高级功能，而是优先保证：

1. 上传稳定。
2. 浏览流畅。
3. 管理方便。
4. 权限安全。
5. 删除可恢复。
6. 容量可统计。
7. 后续可扩展 AI、分享、移动端等能力。

### 2.1 MVP 范围裁剪原则

为避免一期范围过大，MVP 按“先打通个人照片管理闭环，再扩展共享协作”的原则推进。

| 范围 | 必做能力 | 说明 |
|---|---|---|
| MVP 必做 | 注册登录、个人空间、照片上传、照片墙、预览、下载、删除到回收站、恢复、永久删除、基础容量统计 | 保证单用户照片管理闭环可上线 |
| MVP+1 | 共享空间、成员管理、相册、批量操作、搜索筛选、上传失败重试 | 在主流程稳定后补充协作和管理效率 |
| 后续版本 | 分享链接、标签、时间轴、评论、动态、AI、地图、移动端 | 不进入一期核心验收 |

一期验收优先看 MVP 必做能力是否稳定，不以功能数量作为主要判断标准。

### 6.3.1 上传与图片处理策略

MVP 阶段优先采用后端中转上传，便于统一做鉴权、格式校验、容量校验和缩略图生成。后续如果照片数量、单文件体积或并发上传压力上升，再演进为对象存储预签名直传。

推荐规则：

1. 单张照片默认不超过 20MB，单次上传数量需要设置上限，例如 20 张。
2. 后端必须使用真实文件内容或图片解析结果校验类型，不能只信任扩展名和 `Content-Type`。
3. 上传接口需要支持幂等标识，避免前端重试造成重复记录。
4. 缩略图和预览图生成可以 MVP 同步处理，但必须设置超时和失败状态。
5. 后续建议将图片处理改为异步队列，上传成功后照片状态先进入 `processing`，处理完成后变为 `normal`。
6. 如果文件已写入对象存储但数据库写入失败，需要记录并清理孤儿文件。
7. 如果数据库写入成功但图片处理失败，照片记录不能静默丢失，应保留错误状态并支持后台重试。

### 6.5.1 文件存储字段建议

`original_url`、`preview_url`、`thumbnail_url` 可以作为前端读取字段，但不建议只存长期公开 URL。更稳妥的方式是保存对象存储 key，并由后端按权限生成临时访问地址。

如果一期不单独创建 `photo_files` 表，至少需要在 `photos` 表中保留 `storage_key`、`storage_bucket`、`checksum`、`processing_status` 等字段，便于后续做迁移、重试、清理和重复检测。

### 9.4 业务规则

1. 删除照片进入回收站，不立即释放容量。
2. 永久删除后释放容量。
3. 回收站照片默认保留 30 天。
4. 恢复照片后回到原空间。
5. 如果原相册已删除，照片只恢复到空间。
6. 清空回收站需要强确认。
7. 普通成员不能永久删除他人照片。
8. 所有者和管理员可以管理共享空间内回收站照片。

### 9.4.1 容量一致性规则

容量统计需要按强一致口径设计，避免并发上传导致超额或统计错误。

1. 上传容量校验和容量增加必须在同一个事务或原子更新中完成。
2. 并发上传时，以数据库中的 `storage_used + upload_size <= storage_limit` 作为最终判断。
3. 上传失败、图片处理失败或数据库写入失败时，不能增加最终容量。
4. 删除到回收站不释放容量，因为文件仍然占用存储。
5. 永久删除必须先确认文件删除或进入待清理队列，再扣减容量。
6. 容量扣减需要幂等，重复永久删除请求不能重复扣减。
7. 建议增加定期容量重算任务，用照片表和文件表重新校准 `users.storage_used` 与 `spaces.storage_used`。

### 9.5 验收标准

1. 删除后的照片可以在回收站看到。
2. 用户可以恢复照片。
3. 用户可以永久删除照片。
4. 永久删除后不可恢复。
5. 永久删除后容量释放。
6. 容量不足时禁止上传。
7. 用户可以看到个人容量。
8. 空间详情可以看到空间容量。

1. 所有外键关系建议使用显式约束，删除行为需要按业务规则决定，不能默认级联删除照片文件。
2. 相册删除只删除 `albums` 和 `album_photos` 关系，不删除 `photos`。
3. 照片永久删除应先标记为待删除或在事务中记录删除任务，再由文件清理逻辑处理对象存储文件。
4. 对 `storage_used` 这类聚合字段，需要保留可重算来源，避免长期运行后统计漂移。

# 14.1 UI 设计与响应式约束

## 14.1.1 Apple 风格设计原则

产品界面统一采用 Apple 风格，不做传统后台管理系统视觉。

1. 视觉基调：简洁、轻量、高留白、强内容聚焦，避免厚重边框和复杂装饰。
2. 色彩：以浅色背景、柔和灰阶、低饱和强调色为主，使用语义化色彩 token，不在组件中散落原始 hex。
3. 材质：可在导航、浮层、预览面板使用适度毛玻璃效果，但不能影响文字对比度。
4. 圆角和阴影：卡片、弹窗、按钮保持统一圆角体系，阴影轻柔，避免重阴影和廉价玻璃拟态。
5. 字体：优先使用系统 UI 字体栈，中文场景保证可读性；正文基础字号不低于 16px。
6. 图标：使用统一 SVG 图标库，不使用 emoji 作为功能图标。
7. 动效：使用 150ms - 300ms 的轻量过渡，动效服务于层级和状态变化，并支持 `prefers-reduced-motion`。
8. 可访问性：文本对比度、键盘访问、焦点态、aria-label、触控目标尺寸必须满足基础可访问性要求。

## 14.1.2 手机、平板、PC 适配规则

| 设备 | 断点建议 | 布局要求 |
|---|---|---|
| 手机 | `< 768px` | 单列布局，底部或顶部轻导航，照片网格 2-3 列，操作进入底部 sheet |
| 平板 | `768px - 1199px` | 双栏或紧凑侧栏，照片网格 3-5 列，详情面板可抽屉展示 |
| PC | `>= 1200px` | 左侧导航 + 主内容区 + 可选详情侧栏，照片网格 5 列以上 |

适配要求：

1. 必须移动优先设计，不能只做 PC 后压缩到手机。
2. 所有核心页面必须在手机、平板、PC 下可访问、可上传、可预览、可删除和可恢复。
3. 不允许横向滚动作为常规使用方式。
4. 触控目标最小 44px，操作按钮之间至少保留 8px 间距。
5. 手机端批量操作、照片详情、删除确认建议使用 bottom sheet。
6. PC 端可使用侧边栏、快捷键和悬浮工具条提升效率，但不能成为唯一操作入口。
7. 图片列表必须使用响应式尺寸、懒加载和占位比例，避免 CLS。

# 15. 关键技术注意事项

## 15.1 图片不要直接加载原图

照片墙和瀑布流必须加载缩略图，否则页面性能会很差。

推荐策略：

1. 列表页加载 thumbnail_url。
2. 预览页加载 preview_url。
3. 下载时使用 original_url。
4. 原图地址不要长期公开暴露。

## 15.2 权限必须后端校验

前端隐藏按钮只是体验优化，不能作为安全依据。

后端所有接口都需要校验：

1. 用户是否登录。
2. 用户是否为空间成员。
3. 用户是否有当前操作权限。
4. 用户是否可以访问当前照片。
5. 用户是否可以删除或恢复当前照片。

## 15.3 删除不要立即物理删除

照片误删风险高，一期必须做回收站。

推荐规则：

1. 删除时 status = deleted。
2. deleted_at 记录删除时间。
3. deleted_by 记录删除人。
4. 永久删除时再删除文件并释放容量。

## 15.4 容量统计从一期就要做

照片应用存储成本高，必须从第一版开始统计容量。

推荐规则：

1. 上传成功后增加容量。
2. 删除到回收站不释放容量。
3. 永久删除后释放容量。
4. 容量不足时禁止上传。

## 15.5 对象存储建议

本地开发可使用 MinIO，生产环境可使用：

1. 阿里云 OSS。
2. 腾讯云 COS。
3. AWS S3。
4. Cloudflare R2。

## 15.6 安全与稳定性建议

1. 对象存储桶默认私有，所有图片访问都应经过后端权限校验或短期签名 URL。
2. 上传接口需要限制请求体大小、单用户并发数、单空间并发数。
3. 所有高风险操作需要幂等设计，特别是删除、恢复、永久删除和容量扣减。
4. 后端错误响应不要暴露存储 key、内部路径、堆栈信息。
5. 需要记录关键审计日志：登录失败、上传失败、删除、永久删除、成员角色变更。
6. 图片解析库需要限制像素尺寸和处理超时，避免超大图片或恶意图片拖垮服务。

## 15.7 NestJS 与 PostgreSQL 实现约束

1. NestJS 需要按业务域拆模块，避免所有逻辑堆在单个 service 中。
2. Controller 只处理请求入参和响应，不直接写复杂业务逻辑。
3. DTO 必须使用 class-validator / class-transformer 或等价机制做输入校验。
4. 全局启用 ValidationPipe、统一异常过滤器、统一响应错误格式。
5. 数据库迁移必须可重复执行和回滚，不能依赖手工 SQL 修改线上库。
6. PostgreSQL 中角色、状态等字段建议使用 enum 或 check constraint 保证合法值。
7. 容量更新、上传记录写入、永久删除扣减等关键操作必须使用事务或原子 SQL。
8. 常用查询必须有明确索引，并通过 explain 检查照片列表、回收站、空间列表等高频接口。
